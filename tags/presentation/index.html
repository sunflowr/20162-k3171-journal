Module4 presentation

October 27, 2016 · 2 minute read · module4

  • Tags
  • presentation

We had decided on meeting up a few minutes earlier to test things out and calibrate sensors and setup Ableton and Max for Live. I was a bit late because I had problem finding the DVI/HDMI adapter for the computer, but we still managed to do the things we planned.

I think the presentation went fine for most part. I agreed with the feedback that it would have been much better if we had said the text instead of showing it in the video (see Video).

It also seemed like we failed to convey that we tried lots of different combinations of how we used the sensors before ending up were we currently are.

We should have been more clear about important finding such as the breaking point were realized best practice (for us in this special case) for using the accelerometer.

As we didn’t get any real critic of the openness, tightness or ambiguity we assume that we applied it in a good way in the interaction.

Learnings

  • Through the sketching process we realized that the best way to use the accelerometer is to stop thinking of it as absolute values and instead use the shifting values over time to to either increase or decrease something.

  • Another important realisation is that “less is more” still holds true. After presenting the current sketch to the teachers a few days before the presentation - thinking that we had something that felt good - we quickly realized that the use of so many different sensors and sounds in combination created something very complex and hard to control. The only reason we thought it was “ok” was because we were so use to it and knew how it worked. After that we tried simplifying the design but removing sounds and simplifying the usage of the accelerometer.

  • It’s important to setup and explain the limitations (rules) to the person interacting with the artifact - if there’s something that could break it. As an example if the person using the chair didn’t sit on it with the Y-axis of the accelerometer facing forward when sitting down, the readings would be all wrong.

Module3 presentation

October 13, 2016 · 2 minute read · module3

  • Tags
  • presentation

For the presentation we presented how we came to the current design; going from from the initial ideas to how we combined them picking out the essence of what we tried doing - allow the user to move faster through a article based system, something we call fast scroll or filter scroll.

We also talked a bit about how we wanted to incorporate force touch to allow for either interacting with comments within the article or as a way to allow for the activation of the functionality any where on the screen rather than having it region-based.

Presentation

The main critic would probably be around if this would have worked with out the very clearly painted touch regions we used in the presentation. This is something that we should have tested to see if it would have caused confusions.

It could also be argued on how we interpreted nuance, while we certainly allowed for a nuance in they way the person scrolls between texts, it’s not a gradual smooth way. It’s more of a step-based approach.

It would also have been good to test this out in landscape view and evaluate if this is interaction that works in any mode or if the width of the screen makes any difference to the interaction.

Presentation of Module2

September 30, 2016 · 1 minute read · module2

  • Tags
  • presentation

We meet up in the morning to test so everything worked, of course it didn’t. One important wire was broken and needed replacement and one vibrator wire needed to be re-attached.

After we confirmed everything as working we went to the presentation.

Presentation

While a bit nervous the presentation went fine. They seemed to think it was an interesting concept. Nothing really negative to say about it. They suggested taking a step back from our context and see in what other areas/context the idea could be used - to explore outside the context.

They seemed to like the idea of giving one of the user all the control (touch) while the other loosing the control (user with vibrator) having to rely on the other user.

Module 1 final thoughts

September 20, 2016 · 1 minute read · module1

  • Tags
  • status
  • issues
  • presentation

Yesterday was the presentation. We meet up an hour before to sync and it was discovered that there was some issues in the prototype. Apparently something had changed in the data that we hadn’t taken into consideration (a registered device without any discovered devices). After fixing this we had a small sync and some coffee. After that it was time for presentation.

Presentation

While it seemed hard to understand our concept at first, I felt that the audience grasped somewhat in the end what we tried to explore. I guess that we had not found a good way of describing our prototype or the findings. Our concept was a bit abstract and on a meta level where we were more focusing on the idea of space.

We had some critique on the concepts use of bluetooth which I can in hindsight understand, as we focused more on exploring space and how things are interpreted.

Things to consider for the future

  • How do we present the findings?
